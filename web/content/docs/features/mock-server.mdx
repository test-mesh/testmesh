TestMesh includes a built-in mock server that lets you simulate external APIs without external tools. Mock definitions live alongside your tests in YAML, making tests fully self-contained and portable.

## Basic Mock Server

Define mock endpoints directly in your flow's `setup` block. The mock server starts before your steps run and is stopped in `teardown`.

```yaml title="flow-with-mock.yaml"
flow:
  name: "Test with Mock API"

  setup:
    - id: start_mock_server
      action: mock_server_start
      config:
        port: 9000
        endpoints:
          - path: /api/users/123
            method: GET
            response:
              status: 200
              body:
                id: "123"
                name: "John Doe"
                email: "john@example.com"

          - path: /api/users
            method: POST
            response:
              status: 201
              headers:
                Location: "/api/users/124"
              body:
                id: "124"
                message: "User created"
      output:
        mock_url: "$.base_url"

  steps:
    - id: call_api
      action: http_request
      config:
        method: GET
        url: "${start_mock_server.mock_url}/api/users/123"
      output:
        user: "$.body"

  teardown:
    - id: stop_mock_server
      action: mock_server_stop
```

### Loading from an External File

For complex mocks, keep the definition in a separate file and reference it:

```yaml title="mocks/user-service.yaml"
mock_server:
  name: "User Service Mock"
  port: 9001

  endpoints:
    - path: /api/users/:id
      method: GET
      response:
        status: 200
        body:
          id: "{{path.id}}"
          name: "User {{path.id}}"
          email: "user{{path.id}}@example.com"
```

```yaml title="flow.yaml"
setup:
  - id: start_mock
    action: mock_server_start
    config:
      from_file: "mocks/user-service.yaml"
    output:
      mock_url: "$.base_url"
```

## Request Matching

### Path Parameters

Use `:paramName` syntax to capture dynamic path segments, then reference them in responses with `{{path.paramName}}`:

```yaml
endpoints:
  - path: /api/users/:userId/orders/:orderId
    method: GET
    response:
      status: 200
      body:
        user_id: "{{path.userId}}"
        order_id: "{{path.orderId}}"
        status: "completed"
```

### Query Parameters

Match only when specific query params are present:

```yaml
endpoints:
  - path: /api/users
    method: GET
    match:
      query:
        status: "active"
        role: "admin"
    response:
      status: 200
      body:
        users:
          - id: "1"
            name: "Admin User"
```

### Request Headers

Serve different responses based on Authorization headers:

```yaml
endpoints:
  - path: /api/protected
    method: GET
    match:
      headers:
        Authorization: "Bearer secret-token"
    response:
      status: 200
      body:
        message: "Authorized"

  - path: /api/protected
    method: GET
    response:
      status: 401
      body:
        error: "Unauthorized"
```

### Request Body Matching

Match on exact body fields or use JSON path expressions:

```yaml
endpoints:
  - path: /api/login
    method: POST
    match:
      body:
        username: "admin"
        password: "secret"
    response:
      status: 200
      body:
        token: "valid-token-123"

  - path: /api/users
    method: POST
    match:
      body_matches:
        - "$.email contains '@example.com'"
        - "$.age >= 18"
    response:
      status: 201
      body:
        id: "{{random.uuid}}"
        status: "created"
```

### Priority and Fallback

Use `priority` to control which endpoint matches first, and a wildcard fallback for unmatched requests:

```yaml
endpoints:
  - path: /api/users/admin
    method: GET
    priority: 1
    response:
      status: 200
      body:
        id: "admin"
        role: "administrator"

  - path: /api/users/:id
    method: GET
    priority: 2
    response:
      status: 200
      body:
        id: "{{path.id}}"
        role: "user"

  - path: "*"
    method: "*"
    response:
      status: 404
      body:
        error: "Not found"
```

## Response Types

### Dynamic Templates

Use template variables to generate dynamic responses:

```yaml
endpoints:
  - path: /api/time
    method: GET
    response:
      status: 200
      body:
        server_time: "{{timestamp}}"
        request_id: "{{random.uuid}}"
        client_ip: "{{request.ip}}"
```

### Delayed Responses

Simulate slow or variable-latency APIs:

```yaml
endpoints:
  - path: /api/slow-endpoint
    method: GET
    response:
      status: 200
      delay: 5s
      body:
        message: "This took 5 seconds"

  - path: /api/variable-latency
    method: GET
    response:
      status: 200
      delay:
        min: 100ms
        max: 2s
      body:
        message: "Variable latency"
```

### Error and Fault Responses

Simulate failures to test your error handling and retry logic:

```yaml
endpoints:
  - path: /api/failing-endpoint
    method: POST
    response:
      status: 500
      body:
        error: "Internal server error"
        message: "Database connection failed"

  - path: /api/timeout
    method: GET
    response:
      hang: true  # Never responds

  - path: /api/network-error
    method: GET
    response:
      close_connection: true  # Closes connection immediately
```

### Response from File

Serve large datasets or binary files from disk:

```yaml
endpoints:
  - path: /api/large-dataset
    method: GET
    response:
      status: 200
      body_from_file: "mocks/data/large-dataset.json"

  - path: /api/report
    method: GET
    response:
      status: 200
      headers:
        Content-Type: "application/pdf"
      body_from_file: "mocks/data/report.pdf"
      binary: true
```

## Stateful Mocking

### Scenario-Based States

Define named scenarios where each response transitions to the next state:

```yaml
mock_server:
  endpoints:
    - path: /api/counter
      method: GET
      stateful: true
      scenarios:
        - name: "initial"
          response:
            status: 200
            body:
              count: 0
          next_state: "incremented"

        - name: "incremented"
          response:
            status: 200
            body:
              count: 1
          next_state: "incremented_twice"

        - name: "incremented_twice"
          response:
            status: 200
            body:
              count: 2
          next_state: "incremented_twice"
```

### Request Count Based

Vary responses based on how many times an endpoint has been called — useful for simulating rate limiting or flaky services:

```yaml
endpoints:
  - path: /api/rate-limited
    method: GET
    responses:
      - when: "{{request_count <= 3}}"
        status: 200
        body:
          message: "Success"
          remaining: "{{3 - request_count}}"

      - when: "{{request_count > 3}}"
        status: 429
        headers:
          Retry-After: "60"
        body:
          error: "Rate limit exceeded"
```

### Shared State Across Endpoints

Mock a stateful CRUD API where POST creates data that GET can later retrieve:

```yaml
mock_server:
  state:
    users: []
    next_id: 1

  endpoints:
    - path: /api/users
      method: POST
      response:
        status: 201
        body:
          id: "{{state.next_id}}"
          name: "{{request.body.name}}"
        state_changes:
          - action: "append"
            path: "users"
            value:
              id: "{{state.next_id}}"
              name: "{{request.body.name}}"
          - action: "increment"
            path: "next_id"

    - path: /api/users
      method: GET
      response:
        status: 200
        body:
          users: "{{state.users}}"
          total: "{{state.users.length}}"
```

Reset state between test scenarios:

```yaml
steps:
  - id: reset_mock_state
    action: mock_server_reset_state
    config:
      server: "user-service-mock"
      state:
        users: []
        next_id: 1
```

## Advanced Patterns

### Proxy Mode (Partial Mocking)

Override specific endpoints while proxying everything else to a real service:

```yaml
mock_server:
  proxy:
    target: "https://api.real-service.com"
    pass_through: true

  endpoints:
    - path: /api/users/123
      method: GET
      response:
        status: 200
        body:
          id: "123"
          name: "Mocked User"
```

### Recording and Playback

<Tabs items={["Record Mode", "Playback Mode"]}>
  <Tab value="Record Mode">
    Proxy requests to the real API and save responses for later replay:

    ```yaml
    setup:
      - id: start_recording_mock
        action: mock_server_start
        config:
          mode: "record"
          proxy_target: "https://api.real-service.com"
          save_to: "mocks/recorded/user-service.yaml"
    ```
  </Tab>
  <Tab value="Playback Mode">
    Replay previously recorded responses without hitting the real API:

    ```yaml
    setup:
      - id: start_playback_mock
        action: mock_server_start
        config:
          mode: "playback"
          from_file: "mocks/recorded/user-service.yaml"
    ```
  </Tab>
</Tabs>

### Multiple Mock Servers

Run multiple mock servers simultaneously to simulate a full microservices environment:

```yaml
setup:
  - id: start_user_service_mock
    action: mock_server_start
    config:
      name: "user-service"
      port: 8081
      from_file: "mocks/user-service.yaml"

  - id: start_payment_service_mock
    action: mock_server_start
    config:
      name: "payment-service"
      port: 8082
      from_file: "mocks/payment-service.yaml"

steps:
  - id: call_user_service
    action: http_request
    config:
      url: "${start_user_service_mock.base_url}/api/users/123"

  - id: call_payment_service
    action: http_request
    config:
      url: "${start_payment_service_mock.base_url}/api/charge"
```

### Request Verification

Assert that the mock received the expected requests with the correct data:

```yaml
steps:
  - id: call_api
    action: http_request
    config:
      url: "${mock_url}/api/users"
      method: POST
      body:
        name: "John Doe"

  - id: verify_mock_requests
    action: mock_server_verify
    config:
      server: "user-service"
      assertions:
        - path: /api/users
          method: POST
          count: 1
          body_matches:
            - "$.name == 'John Doe'"
```

### Chaos Engineering

Inject random failures to test resilience:

```yaml
mock_server:
  chaos:
    enabled: true

  endpoints:
    - path: /api/users
      method: GET
      chaos_rules:
        - probability: 0.1
          response:
            status: 500
            body:
              error: "Random server error"

        - probability: 0.05
          response:
            hang: true

        - probability: 0.85
          response:
            status: 200
            body:
              users: []
```

## Template Variables

Available in all mock response templates:

| Variable | Description |
|---|---|
| `{{timestamp}}` | Current ISO 8601 timestamp |
| `{{timestamp_unix}}` | Unix timestamp |
| `{{random.uuid}}` | Random UUID v4 |
| `{{random.int}}` | Random integer |
| `{{random.int(min, max)}}` | Random integer in range |
| `{{random.string(length)}}` | Random alphanumeric string |
| `{{path.paramName}}` | Path parameter value |
| `{{query.paramName}}` | Query parameter value |
| `{{header.HeaderName}}` | Request header value |
| `{{request.body.fieldName}}` | Request body field |
| `{{request.ip}}` | Client IP address |
| `{{request_count}}` | Number of times this endpoint was called |
| `{{state.fieldName}}` | State variable value |

## Full Example: Payment Gateway Mock

```yaml title="payment-gateway-test.yaml"
flow:
  name: "Test Fare Calculation with Mock Payment Gateway"

  setup:
    - id: start_payment_gateway_mock
      action: mock_server_start
      config:
        name: "payment-gateway"
        port: 9000
        endpoints:
          - path: /v1/charges
            method: POST
            match:
              body_matches:
                - "$.amount <= 10.00"
            response:
              status: 200
              delay: 500ms
              body:
                id: "{{random.uuid}}"
                status: "succeeded"
                amount: "{{request.body.amount}}"

          - path: /v1/charges
            method: POST
            match:
              body_matches:
                - "$.amount > 10.00"
            response:
              status: 402
              body:
                error:
                  code: "card_declined"
                  message: "Amount exceeds limit"
      output:
        payment_gateway_url: "$.base_url"

  steps:
    - id: send_taps
      action: kafka_publish
      config:
        topic: "matches"
        value:
          pan: "4111111111111111"
          amount: 5.00

    - id: wait_for_payment
      action: kafka_consume
      config:
        topic: "payments"
        timeout: 10s

    - id: verify_charge_request
      action: mock_server_verify
      config:
        server: "payment-gateway"
        assertions:
          - path: /v1/charges
            method: POST
            count: 1
            body_matches:
              - "$.amount == 5.00"
              - "$.currency == 'USD'"

  teardown:
    - id: stop_mock
      action: mock_server_stop
      config:
        server: "payment-gateway"
```

<Callout type="tip">
  Place mock definitions in a `mocks/` directory alongside your flow files and load them with `from_file`. This keeps flows readable while allowing complex mock configurations to be shared across multiple test flows.
</Callout>
