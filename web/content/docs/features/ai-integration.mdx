## Generate Command

Generate a complete test flow from a natural language description:

```bash
testmesh generate "Test user registration API"
testmesh generate "Test payment gateway with retry on failure" -o flows/payment/
testmesh generate "Test daily fare cap with 3 taps and $5 limit" --provider anthropic
```

**What gets generated:**

```
Generating test flow from description...

  Analyzed intent: Integration test for user registration
  Identified protocols: HTTP (API), Database (PostgreSQL)
  Selected patterns: API test, database validation, cleanup
  Generated flows/user-registration-test.yaml (45 lines)
  Generated data/user-test-data.json (12 users)

Preview:
  Steps:
    1. Create test user via POST /api/users
    2. Verify response (status 201, user ID exists)
    3. Query database to confirm user created
    4. Cleanup test data

Run test now? [y/n]
```

**Example generated output:**

```yaml title="flows/user-registration-test.yaml"
flow:
  name: "User Registration Test"
  description: "Test POST /api/users endpoint for user registration"
  tags: ["api", "users", "registration"]

  env:
    API_URL: "${API_BASE_URL}/api"

  setup:
    - id: cleanup_test_user
      action: database_query
      config:
        query: "DELETE FROM users WHERE email = $1"
        params: ["test@example.com"]
      on_error: "continue"

  steps:
    - id: register_user
      action: http_request
      config:
        method: POST
        url: "${API_URL}/users"
        headers:
          Content-Type: "application/json"
        body:
          email: "test@example.com"
          password: "SecurePass123!"
          name: "Test User"
      output:
        user_id: "response.body.id"
      assert:
        - status_code == 201
        - response.body.id exists
        - response.body.email == "test@example.com"

    - id: verify_in_database
      action: database_query
      config:
        query: "SELECT * FROM users WHERE id = $1"
        params: ["${register_user.user_id}"]
      assert:
        - result.count == 1
        - result.rows[0].email == "test@example.com"

  teardown:
    - id: cleanup
      action: database_query
      config:
        query: "DELETE FROM users WHERE id = $1"
        params: ["${register_user.user_id}"]
      on_error: "continue"
```

## Import from Specifications

### From OpenAPI / Swagger

```bash
testmesh import openapi swagger.yaml
```

```
Analyzing OpenAPI specification...
  API: User Service v2.1
  Base URL: https://api.example.com/v2
  Endpoints: 23

Generating tests...
  GET  /users                → flows/get-users-test.yaml
  POST /users                → flows/create-user-test.yaml
  GET  /users/{id}           → flows/get-user-by-id-test.yaml
  PUT  /users/{id}           → flows/update-user-test.yaml
  DELETE /users/{id}         → flows/delete-user-test.yaml
  ...

Generated 23 test flows, 1 mock server, test data for all scenarios.
```

For each endpoint, TestMesh generates a flow with realistic request data, appropriate assertions, and an optional mock server configuration:

```yaml title="flows/create-user-test.yaml"
flow:
  name: "POST /users - Create User"
  description: "Auto-generated from OpenAPI spec"

  steps:
    - id: create_user
      action: http_request
      config:
        method: POST
        url: "${API_URL}/users"
        body:
          email: "{{faker.email}}"
          name: "{{faker.name}}"
      assert:
        - status == 201
        - response.body matches_schema:
            type: object
            required: [id, email, name]
```

## Configuration

### CLI Configuration

```yaml title=".testmesh/config.yaml"
ai:
  provider: "anthropic"

  providers:
    anthropic:
      api_key: "${ANTHROPIC_API_KEY}"
      model: "claude-sonnet-4-5"
      max_tokens: 8000
      temperature: 0.7

    openai:
      api_key: "${OPENAI_API_KEY}"
      model: "gpt-4"
      max_tokens: 8000

    local:
      host: "localhost:11434"  # Ollama

  generation:
    include_comments: true
    include_examples: true
    max_steps: 50

  analysis:
    coverage_threshold: 0.8
    suggest_improvements: true
```

<Callout type="warning">
  Never commit API keys to version control. Use environment variables (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`) or a secrets manager.
</Callout>

<Callout type="tip">
  Generated flows are always validated against the TestMesh YAML schema before being written to disk. If generation produces an invalid flow, TestMesh will report the issue rather than saving a broken file.
</Callout>

### Workspace-Level Provider Routing

The API server supports workspace-scoped AI providers and per-agent overrides. This lets different workspaces use different LLM providers, and different agents within a workspace use different models.

**Resolution order:**

```
Agent-specific override → Workspace default → Global default → Environment variable
```

For example, route the `generation` agent to Anthropic (better structured output) and the `coverage` agent to OpenAI (lower cost):

```bash
curl -X PUT http://localhost:5016/api/v1/workspaces/$WORKSPACE_ID/ai-config \
  -H "Content-Type: application/json" \
  -d '{
    "default_provider": "anthropic-integration-uuid",
    "agent_overrides": [
      { "agent_name": "coverage", "integration_id": "openai-integration-uuid" }
    ]
  }'
```

See [Workspace Integrations](/docs/features/workspace-integrations) for full details on managing providers and routing.

## Semantic Search

When an OpenAI integration is configured, TestMesh generates vector embeddings for graph nodes, flows, and code changes. AI agents automatically use these embeddings to:

- **Coverage agent**: Detect near-duplicate flows before suggesting new tests
- **Impact agent**: Find semantically related nodes beyond explicit graph edges
- **Diagnosis agent**: Search for past fixes to similar failures
- **Diff analyzer**: Discover affected flows via semantic similarity when tags are incomplete

See [Semantic Search & Embeddings](/docs/features/embeddings) for setup and details.
